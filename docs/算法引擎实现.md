# 算法引擎实现说明

本文档介绍 `game-ai-engine` 服务的完整实现，涵盖对抗环境设计、神经网络结构、IPPO 训练算法、奖励塑形机制以及对外 HTTP 接口。

---

## 1. 整体架构

```
game-ai-engine/
├── app/
│   ├── main.py                  # FastAPI 应用入口
│   ├── api/
│   │   ├── train.py             # 训练控制接口（启动 / 暂停 / 恢复 / 终止）
│   │   ├── infer.py             # 单步推理接口（加载权重，输出动作）
│   │   └── eval_rollout.py      # 批量评估接口（固定种子多局 rollout）
│   ├── core/
│   │   └── trainer.py           # TrainerManager：多任务线程管理 + Redis 发布
│   ├── environments/
│   │   └── grid_env.py          # GridWarParallelEnv：多智能体网格对抗环境
│   └── rl_algorithms/
│       ├── actor_critic.py      # ActorCritic 神经网络
│       └── ippo_trainer.py      # IPPOTrainer：独立 PPO 训练器
```

服务以 **FastAPI + Uvicorn** 运行，训练任务在独立守护线程中执行，进度指标通过 **Redis Pub/Sub** 实时推送给 Java 后端，再经 WebSocket 转发至前端。

## 2. 对抗环境（GridWarParallelEnv）

### 2.1 场景参数

环境由 `GridWarParams` 描述，字段包括：

| 字段 | 含义 | 默认值 |
|------|------|--------|
| `width` / `height` | 网格地图尺寸 | 12 × 12 |
| `red` / `blue` | 红蓝双方兵力数 | 各 2 |
| `obstacles` | 障碍物坐标列表 | 空 |
| `max_steps` | 每局最大步数 | 200 |

场景配置由 `parse_scene_to_params()` 从前端传入的 JSON 解析而来，支持 `red_count` / `red_drone_count` 等多种字段名。

### 2.2 观测空间

每个智能体的观测是以自身为中心的 **5×5 局部栅格**，展平为 **25 维**浮点向量。栅格中每格编码一个语义值：

| 值 | 含义 |
|----|------|
| 0.00 | 空地 |
| 0.25 | 障碍物（含地图边界） |
| 0.50 | 友方单位 |
| 0.75 | 敌方单位 |
| 1.00 | 自身 |

超出地图边界的格子也被编码为 0.25（障碍），保持观测维度固定不变。

### 2.3 动作空间

5 个离散动作：`0=不动, 1=上, 2=下, 3=左, 4=右`。

撞墙或撞障碍时保持原位，不触发错误。

### 2.4 并行决策模型

所有存活智能体在**同一时间步内同时**执行动作，无先后顺序。抢占同一格时随机保留一方，其余退回原位。

### 2.5 战斗机制

当不同队伍的智能体在移动后占据同一格时触发交战：每对红蓝单位各随机选一个，以 **50% 概率**淘汰其中一方。淘汰方获得 `-1.0` 惩罚，对方队伍所有存活成员获得 `+0.3` 奖励。

### 2.6 终止条件

- 一方全部阵亡 → 对方胜利，存活方每人获得 `+2.0` 奖励
- 达到 `max_steps` → 平局，无额外奖励

## 3. 奖励设计

每步奖励由四部分叠加：

```
R_t = R_shaping + R_combat + R_terminal + R_step
```

| 分量 | 值 | 触发条件 |
|------|------|----------|
| 接近奖励（shaping） | +0.02 | 本步移动后与最近敌方曼哈顿距离缩短 |
| 远离惩罚（shaping） | −0.01 | 本步移动后与最近敌方距离增大 |
| 击杀奖励 | +0.3（友方全员） | 本方单位在交战中消灭敌方 |
| 阵亡惩罚 | −1.0（自身） | 本单位在交战中被淘汰 |
| 胜利奖励 | +2.0（存活成员） | 本方获胜 |
| 步骤惩罚 | −0.003（存活单位） | 每步均有 |

**接近/远离奖励（Reward Shaping）** 是解决稀疏奖励的关键。原始环境中胜负奖励仅在对局结束时给出，智能体在地图上很难学到"主动靠近敌人"这一前提行为。引入 shaping 后，每步移动立刻产生梯度，引导策略从随机游走收敛至有目的性的进攻行为。

## 4. 神经网络（ActorCritic）

采用**共享输入、分离头部**的 Actor-Critic 结构：

```python
# Actor（策略头）
pi: Linear(25→128) → Tanh → Linear(128→128) → Tanh → Linear(128→5)

# Critic（价值头）
v:  Linear(25→128) → Tanh → Linear(128→1)
```

- 输入：25 维观测向量
- Actor 输出：5 个动作的 logits，经 `Categorical` 分布采样得到离散动作
- Critic 输出：标量状态价值估计 V(s)
- 激活函数选用 `Tanh`，对 `-1~1` 归一化后的栅格值效果优于 ReLU

## 5. IPPO 训练算法

### 5.1 独立 PPO（IPPO）思路

红蓝双方各维护一个独立的 `ActorCritic` 网络和 `Adam` 优化器，训练时**互不共享参数**，仅通过环境交互产生对抗。这种做法称为 Independent PPO（IPPO），实现简单且在对称对抗场景中效果良好。

### 5.2 训练主循环

```
for epoch in 1..max_epochs:
    # 1. 数据收集
    for _ in rollout_episodes:
        red_buffers, blue_buffers, winner = collect_episode()

    # 2. PPO 更新（红蓝各一次）
    loss_red  = ppo_update(policy_red,  opt_red,  red_buffers)
    loss_blue = ppo_update(policy_blue, opt_blue, blue_buffers)

    # 3. 评估胜率（贪心策略，固定种子 6 局）
    eval_win_rate = eval_win_rate(n_games=6)

    # 4. 指标推送至 Redis
    publish(channel, {epoch, red_win_rate, loss, reward, ...})

    # 5. 每 10 轮保存 checkpoint
```

### 5.3 轨迹收集（collect_episode）

每局 episode：

1. 调用 `env.reset()` 重置地图，智能体随机分布于空闲格子
2. 循环执行决策步：所有存活智能体调用各自策略的 `act(obs)` 采样动作，同时记录 `log_prob` 和 `value`
3. 调用 `env.step(actions)` 推进环境，收集奖励和 done 标志
4. 对局结束（胜负或超时）后返回各智能体的 `RolloutBuffer`

`RolloutBuffer` 记录：`obs, actions, rewards, dones, log_probs, values`

### 5.4 PPO 更新（_ppo_update）

```
对所有 buffer 中的轨迹合并:
  ret = discount_returns(rewards, dones, γ)   # 折扣累计回报

对全量数据做 K 轮（ppo_epochs）mini-batch 更新:
  adv = ret - V(obs)                          # 优势估计
  adv = (adv - mean) / (std(unbiased=False) + ε)  # 标准化

  ratio = exp(log_π(a|s) - log_π_old(a|s))   # 重要性权重
  surr1 = ratio × adv
  surr2 = clamp(ratio, 1-ε, 1+ε) × adv
  L_pi  = -min(surr1, surr2).mean()           # PPO-Clip 策略损失

  L_v   = MSE(V(obs), ret)                    # 价值损失
  L_ent = -entropy.mean()                     # 熵奖励（鼓励探索）

  loss = L_pi + vf_coef × L_v + ent_coef × L_ent
  梯度裁剪（max_grad_norm）后 optimizer.step()
```

`std(unbiased=False)` 使用总体标准差，避免单样本 batch 时自由度为 0 触发 PyTorch 警告。

### 5.5 推荐超参数

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| `learning_rate` | 1e-3 | Adam 学习率 |
| `gamma` | 0.99 | 折扣因子 |
| `clip_coef` | 0.2 | PPO-Clip ε |
| `vf_coef` | 0.5 | 价值损失权重 |
| `ent_coef` | 0.05 | 熵奖励权重（提升探索） |
| `rollout_episodes` | 20 | 每轮收集的 episode 数 |
| `ppo_epochs` | 6 | 每批数据的更新轮次 |
| `max_epochs` | 500 | 总训练轮数 |

## 6. 训练任务管理（TrainerManager）

`TrainerManager` 负责多任务并发管理：

- 每个训练任务以独立 **守护线程** 运行，互不阻塞
- 每个任务对应一个 `JobContext`，包含两个 `threading.Event`：
  - `pause`：置位后训练循环在 epoch 入口自旋等待
  - `stop`：置位后训练循环在下一个检查点提前退出
- 训练器连接 Redis，每个 epoch 结束后向频道 `channel:train:metrics:{task_id}` 发布指标 JSON，Java 后端订阅后通过 WebSocket 推送至前端

## 7. HTTP 接口

### 训练控制

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/train/start` | 启动训练任务（在新线程中异步运行） |
| POST | `/api/train/pause` | 暂停指定任务 |
| POST | `/api/train/resume` | 恢复指定任务 |
| POST | `/api/train/stop` | 终止指定任务 |

`/api/train/start` 请求体：
```json
{
  "task_id": "uuid",
  "scene_config": { "map_width": 8, "map_height": 8, "red_count": 2, "blue_count": 2 },
  "algo_config":  { "learning_rate": 0.001, "max_epochs": 500, ... }
}
```

### 单步推理

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/infer/predict` | 加载 checkpoint，对单帧观测输出动作 |

请求体：
```json
{
  "checkpoint_dir": "checkpoints/uuid",
  "team": "red",
  "observation": [0.0, 0.25, ..., 1.0]  // 25 维
}
```
返回：`{ "action": 2, "value": 0.38, "logits": [...] }`

### 批量评估

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/eval/rollout` | 固定种子多局贪心 rollout，返回胜率统计 |

返回：
```json
{
  "n_episodes": 16,
  "red_wins": 9,
  "blue_wins": 5,
  "draws": 2,
  "red_win_rate": 0.5625,
  "mean_red_return": 1.23
}
```

## 8. Checkpoint 机制

训练过程中每隔 10 个 epoch 及最终轮次保存权重：

```
checkpoints/{task_id}/
├── policy_red_ep10.pt
├── policy_blue_ep10.pt
├── policy_red_ep20.pt
├── ...
├── policy_red_final.pt   ← 推理接口和评估接口加载此文件
└── policy_blue_final.pt
```

权重通过 `torch.save(state_dict, path)` 存储，加载时指定 `map_location` 保证 CPU/GPU 跨设备兼容。
