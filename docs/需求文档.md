## 2. 总体描述 (Overall Description)

### 2.1 产品视角
[cite_start]本系统采用 B/S 架构设计 [cite: 6][cite_start]，逻辑上划分为前端交互展示层（Vue.js）、后端业务控制层（Java 等）以及底层数据存储层（MySQL 等）[cite: 7]。系统作为智能体训练的综合管理中枢，向下调度分布式算法引擎，向上为指挥官提供可视化决策支持界面。

### 2.2 用户特征分析
* **系统管理员：** 负责平台基础数据的维护、用户权限管理及系统日志监控。具备一定的 IT 运维能力。
* **算法工程师：** 负责数据接入、算法超参数配置及智能体训练任务的启停。熟悉深度强化学习原理。
* **战术指挥官（最终用户）：** 负责查看智能体性能对比分析报告，提取最优战术策略并应用于实际或推演场景。主要关注态势可视化与结果准确性。

### 2.3 运行环境约束
* **客户端：** 支持主流现代浏览器（Chrome 90+, Edge 等），无须安装额外插件。
* **服务端：** * 操作系统：Linux (CentOS 7+ 或 Ubuntu 20.04+)。
  * 运行环境：JDK 17+, Python 3.9+。
  * [cite_start]数据库：MySQL 8.0+ 或 Oracle [cite: 7]。

## 3. 具体功能需求 (Functional Requirements)

[cite_start]本节详细描述系统的五大核心业务模块 [cite: 19]。

### [cite_start]3.1 博弈对抗场景配置模块 [cite: 20]
[cite_start]**功能描述：** 提供可视化的军事红蓝双方对抗环境定义功能 [cite: 26]。
* **FR-1.1 战场地图设置：** 支持用户设定二维/三维网格地图的尺寸（如 100x100 矩阵），并可自定义障碍物（如山地、禁飞区）的位置坐标。
* [cite_start]**FR-1.2 兵力部署配置：** 支持定义红蓝双方无人机集群的初始数量 [cite: 26]、初始坐标位置及性能参数（如速度、探测半径）。
* **FR-1.3 任务目标定义：** 支持配置当前博弈场景的胜负判定条件（如：全歼敌方、占领特定区域持续 N 个时间步等）。

### [cite_start]3.2 数据接入管理模块 [cite: 21]
[cite_start]**功能描述：** 负责处理和清洗战场环境作战单元以往动作的价值分析数据 [cite: 18]。
* **FR-2.1 外部数据源挂载：** 支持通过 API 接口或文件上传（CSV/JSON）的方式，接入历史作战单元动作数据。
* [cite_start]**FR-2.2 态势价值解析：** 系统需能解析并持久化每一个时间节点特定动作对整体战场态势的价值评估参数 [cite: 18]。

### [cite_start]3.3 算法配置模块 [cite: 22]
**功能描述：** 提供灵活的强化学习模型超参数配置面板。

* **FR-3.2 超参数动态调节：** 提供表单界面供用户配置关键参数，包括但不限于：学习率（Learning Rate）、折扣因子（Discount Factor）、探索衰减率（Epsilon Decay）及最大训练轮次（Epochs）。
* **FR-3.3 配置模板管理：** 支持将当前调优好的超参数组合保存为模板，方便一键复用。

[cite_start]**功能描述：** 训练任务的核心编排与执行调度枢纽，支持多个智能体的大规模并行训练 [cite: 17]。
* **FR-4.1 任务生命周期管理：** 提供训练任务的“创建、排队、启动、暂停、恢复、终止”全流程控制。
* **FR-4.2 分布式调度支持：** 系统后端能够根据算法配置，向底层的 Python 训练引擎下发并行计算指令。
* **FR-4.3 实时状态监控：** 实时监听训练进程，防范并记录资源溢出（OOM）或死锁等异常状态。

## 4. 非功能性需求 (Non-Functional Requirements)

### 4.1 性能需求
* **响应时间：** 前端页面常规操作（如场景配置保存）响应时间应 $\le$ 1.5秒；实时监控大屏的数据刷新延迟应 $\le$ 2秒。
* [cite_start]**并发处理：** 系统后端应能稳定支撑多个大规模并行训练任务 [cite: 17] 的同时挂载与状态推送，不出现请求阻塞。

### 4.2 可靠性与可用性
* **异常恢复：** 当底层 Python 训练脚本因意外崩溃时，Java 编排控制台应能捕获异常，将任务状态标记为“失败”并释放数据库及内存资源，防止平台整体宕机。
* **数据持久化：** 训练过程中的关键指标（Loss, Reward）需定期落盘至 MySQL，防止断电导致整个训练周期的心血丢失。

### 4.3 界面与易用性需求
* 界面采用响应式设计，主色调建议采用契合军工/航天背景的深色系（如深蓝、暗灰），以突出“态势感知”的专业感。
* 所有复杂表单（如算法超参数配置）必须提供默认推荐值及字段级的悬浮提示（Tooltip）帮助。
